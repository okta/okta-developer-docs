Secure AWS Bedrock Agents with Okta
Learning outcomes
Complete the Okta setup and add the shared token exchange module to your project.
Set up an AWS Bedrock Agent with action group Lambda functions that can call Okta-protected APIs.
Invoke the Bedrock Agent with the user's Okta identity passed as sessionAttributes.
Test the end-to-end flow locally and verify that action group Lambdas receive the access_token.
What you need
Okta setup complete (see Token exchange reference); have your AGENT_CLIENT_ID, AGENT_KEY_ID, and AGENT_PRIVATE_KEY_JWK ready
AWS account with Amazon Bedrock available in your region
AWS CLI configured (aws configure sso or access key + secret)
Python 3.10+

How it works
Your calling application performs the Okta token exchange, then passes the resulting access_token and user claims to the Bedrock Agent as sessionAttributes; action group Lambdas read the token from there to call Okta-protected APIs. For a full explanation of the token exchange, see Secure imported AI agents.

Invoke the Bedrock Agent
After the token exchange, invoke the agent and pass the access_token and user claims as sessionAttributes:
python
import boto3, uuid
from token_exchange import get_id_jag, get_access_token

def invoke_bedrock_agent(prompt, user_claims, access_token):
    client = boto3.client("bedrock-agent-runtime", region_name=AWS_REGION)

    response = client.invoke_agent(
        agentId=BEDROCK_AGENT_ID,
        agentAliasId=BEDROCK_AGENT_ALIAS_ID,
        sessionId=str(uuid.uuid4()),  # reuse across turns for multi-turn sessions
        inputText=prompt,
        sessionState={
            "sessionAttributes": {
                "okta_access_token": access_token,
                "user_name":  user_claims.get("name", ""),
                "user_email": user_claims.get("email", ""),
                "user_sub":   user_claims.get("sub", ""),
            }
        }
    )

    # Response is a streaming EventStream
    chunks = [
        e["chunk"]["bytes"].decode()
        for e in response["completion"]
        if "chunk" in e
    ]
    return "".join(chunks)

V1
Secure AWS Bedrock Agents with Okta
AWS Bedrock Agents is Amazon's managed AI agent service; this guide shows how to call it securely using Okta's token exchange. Your application exchanges the user's id_token for a scoped access_token through Okta, then invokes the Bedrock Agent with the user's identity passed as session attributes so that action group Lambda functions can call Okta-protected APIs on the user's behalf.
How it works
The flow has two stages. First, your application performs Okta's two-step token exchange: the user's id_token becomes an ID-JAG at the Org Authorization Server, then the ID-JAG becomes a scoped access_token at the Custom Authorization Server. Second, your application invokes the Bedrock Agent and passes the access_token and user claims as sessionAttributes.
User
  { "prompt": "...", "id_token": "<okta_id_token>" }
  ↓
Application (main.py)
  Step 1: id_token → ID-JAG      (Org AS — /oauth2/v1/token)
  Step 2: ID-JAG  → access_token (Resource AS — /oauth2/{custom-as}/v1/token)
  ↓
AWS Bedrock Agent  (bedrock-agent-runtime · invoke_agent)
  sessionAttributes: { okta_access_token, user_name, user_email, user_sub }
  ↓
Action Group Lambda
  Authorization: Bearer <access_token>
  ↓
Okta-protected API
Key characteristic: no gateway, no interceptor. The calling application owns the full Okta token exchange; the Bedrock Agent receives a ready-to-use access_token in sessionAttributes.

Part 1: Configure Okta
These steps are performed once per Okta org by an admin. Collect the values noted here; you will use them as environment variables in Part 3.
1. Create an OIDC web app
This app handles user-facing login and issues the id_token your application exchanges.
In the Okta Admin Console, go to Applications > Create App Integration.
Select OIDC - OpenID Connect, then Web Application.
Set the following:
Grant types: Authorization Code
Sign-in redirect URI: your application's callback URL (for local testing: http://localhost:5000/callback)
Scopes: openid profile email
Save. Note the Client ID and Client Secret.
2. Set up a Custom Authorization Server
Go to Security > API > Authorization Servers.
Use the built-in default server or create a new one.
Note the Issuer URL (for example: https://<your-domain>/oauth2/default).
3. Add a custom scope
System scopes (openid, profile, email) are stripped during the ID-JAG exchange and cause an invalid_scope error. You must create a custom scope instead.
In your Custom AS, go to Scopes > Add Scope.
Add the scope: xaa:read
4. Create an AI Agent (WORKLOAD type)
This is the machine identity your application uses to sign token exchange requests.
In the Admin Console, go to AI Agents > Create new agent.
Under Client Authentication, select Public Key / Private Key (private_key_jwt).
Generate an RSA keypair. Register the public key JWK with the agent.
Under Connected Resource:
Link to the OIDC web app from step 1.
Set the Custom AS to the one from step 2.
Allowed scopes: include xaa:read.
Set Status to Active.
Save. Note the Agent Client ID, the key kid, and keep the private key JWK (shown once only).
5. Configure the access policy
In your Custom AS, go to Access Policies.
Edit the default rule or create a new one.
Enable grant type: JWT Bearer (urn:ietf:params:oauth:grant-type:jwt-bearer).
Add the AI Agent as an allowed client.

Token exchange flow
The application makes two HTTP POST calls to Okta's token endpoints directly — no Okta SDK is required.
Step 1: Exchange id_token for ID-JAG (Org AS)
POST /oauth2/v1/token
grant_type=urn:ietf:params:oauth:grant-type:token-exchange
client_assertion_type=urn:ietf:params:oauth:client-assertion-type:jwt-bearer
client_assertion=<signed JWT, aud=org token URL>
subject_token=<user id_token>
subject_token_type=urn:ietf:params:oauth:token-type:id_token
requested_token_type=urn:ietf:params:oauth:token-type:id-jag
scope=xaa:read
audience=https://<domain>/oauth2/<custom-as-id>
Step 2: Exchange ID-JAG for access_token (Custom AS)
POST /oauth2/<custom-as-id>/v1/token
grant_type=urn:ietf:params:oauth:grant-type:jwt-bearer
client_assertion_type=urn:ietf:params:oauth:client-assertion-type:jwt-bearer
client_assertion=<signed JWT, aud=custom AS token URL>
assertion=<id_jag from Step 1>
The client_assertion is a short-lived JWT (5-minute expiry) signed with the agent's RSA private key using RS256. The kid header must match the registered public key.
{
  "iss": "<agent_client_id>",
  "sub": "<agent_client_id>",
  "aud": "<token_endpoint_url>",
  "iat": <now>,
  "exp": <now + 300>,
  "jti": "<uuid>"
}
